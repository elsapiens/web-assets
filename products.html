Item Master Overview

The Item Master contains three types of item definitions. Each serves a different purpose.

👕

Items

Standalone products and services

👔👕

Item Groups

Variants (size, color, etc.)

🎁

Composite Items

Bundles sold as one unit

Key Difference Item Groups = same item, different attributes (T-shirt S/M/L). Composite Items = different items bundled (T-shirt + Shorts + Bag).
1.1
Items (Standalone)

Items are the basic unit. Each represents a single product or service.

Goods (Products)

Physical items with inventory tracking. Has stock, can be stored, valued using FIFO/Weighted Avg.

E.g., Laptop, T-Shirt, Coffee

Services

Non-physical items, no inventory. Delivered, not stored. Time or task based.

E.g., Installation, Consulting, Repair
1.2
Item Groups (Variants)

Create multiple variants of the same item with attributes like size, color, material.

Without Item Groups

  • T-Shirt Small Blue
  • T-Shirt Small Red
  • T-Shirt Medium Blue
  • ... 12+ items to manage

With Item Groups

Group: T-Shirt
Size: S, M, L, XL Color: Blue, Red

System generates all variants automatically

Variant Pricing All variants inherit the base price. Override individual variants when needed (e.g., XL +$3, Premium color +$10).
1.3
Composite Items

Group different items together and sell as a single unit. Two types based on inventory tracking.

Assembly Item

Items combined into a single unit. Inventory tracked at assembled level.

Example: Desktop Computer
CPU + RAM + HDD → Build → Track "Computer" stock

Kit Item

Items sold together but remain separate. Inventory at component level.

Example: Gift Basket
Wine + Cheese + Crackers → Each reduces on sale
ScenarioTypeWhy
Manufacturing PCs from partsAssemblyComponents combined, can't separate
Gift basket with wine + cheeseKitItems can be sold separately too
Furniture set (table + chairs)KitEach piece has own identity
Associated Services Composite Items can include services (e.g., Washing Machine + Free Installation). Track cost even for free services to calculate true margins.

4
Financial Parameters

Items are commercial definitions. The transaction context determines how they're treated financially. This is event-driven accounting.

The Core Principle

The same item can be Inventory, Expense, or Fixed Asset - depending on how it's used. The item doesn't decide accounting; transactions do.

Inventory

Purchased for resale

Expense

Consumed in operations

Fixed Asset

Long-term internal use

Event-Driven Architecture: How It Works

Items hold parameters. Events read those parameters and create journal entries. The journal is the source of truth.

ITEM
Holds parameters
valuation method, HSN, etc.
→
EVENT
Triggers processing
purchase, sale, transfer
→
JOURNAL ENTRY
Source of truth
Dr/Cr accounts

Purchase Event

  1. PO received → triggers purchase event
  2. System reads item's valuation method
  3. System reads item's HSN → determines tax
  4. Cost layers updated (FIFO) or average recalculated
  5. Journal entry created:
    Dr Inventory | Cr Accounts Payable

Sale Event

  1. Invoice created → triggers sale event
  2. System reads item's valuation method
  3. COGS calculated (FIFO: oldest layer, or avg cost)
  4. Store's price list applied → selling price
  5. Journal entries created:
    Dr COGS | Cr Inventory
    Dr Receivable | Cr Revenue
Key Insight Items don't "do" accounting. Items are data. Events read item data, apply policies, and create journal entries. The sections below define the parameters that events consume.
4.1
Permission Flags

Each item has permission flags that control where it appears in the system.

can_sell = TRUE

Item appears in Sales screens, can be added to sales orders

can_purchase = TRUE

Item appears in Purchase screens, can be added to purchase orders

Example: "Company Laptop" might have can_purchase = TRUE (IT can buy it) but can_sell = FALSE (you don't sell laptops to customers).

4.4
Inventory Valuation

Determines how you calculate the cost of items in stock. Each item has ONE valuation method.

FIFO (First In, First Out)

Oldest inventory sold first. COGS reflects older prices.

Jan: 10 @ $100
Feb: 10 @ $120
Sell 5: 5 × $100 = $500
Best for: Perishables, dated inventory

Weighted Average

All units valued at average cost. Each purchase updates average.

Jan: 10 @ $100
Feb: 10 @ $120
Avg: $110
Sell 5: 5 × $110 = $550
Best for: Commodities, homogeneous items
One Method Per Item Cannot use FIFO for one store and Weighted Average for another. Must be consistent across all locations for accurate accounting.
4.5
Fixed Assets & Depreciation

When items are purchased/produced for long-term internal use (not resale), they become Fixed Assets that lose value over time through depreciation.

Asset Categories

Define default depreciation rules for groups of similar assets:

CategoryUseful LifeSalvageMethodFrequency
Office Furniture7 years10%Straight LineMonthly
Computer Equipment3 years$0Straight LineMonthly
Vehicles5 years15%Declining BalanceMonthly
Machinery10 years5%Straight LineMonthly
Buildings30 years20%Straight LineAnnually

Depreciation Methods

Straight Line

Same amount every year. Simple and predictable.

(Cost - Salvage) / Useful Life
Example: $10,000, $1,000 salvage, 5 years
($10,000 - $1,000) / 5 = $1,800/year

Declining Balance

Higher depreciation early, less later. Based on remaining book value.

Book Value × Rate (Rate = 2 / Life)
Example: $10,000, 5 years, 40% rate
Y1: $4,000 | Y2: $2,400 | Y3: $1,440...
Why Categories Matter: Without categories, users manually enter parameters for every asset. Categories enforce consistency while allowing overrides when needed.

4.7
Key Takeaways
  • Items are simple definitions - they don't contain accounting rules or fixed prices
  • Context determines treatment - same item can be expense, inventory, or asset
  • Flags control availability - can_sell and can_purchase determine where items appear
  • Price Lists have 3 methods - Markup, Markdown, and Tiered (quantity-based)
  • Multiple price lists per item - retail, wholesale, employee, clearance, etc.
  • Discounts apply on top - store-level promotions after price list calculation
  • Asset Categories define defaults - depreciation rules can be overridden per asset

Stores

Stores are your selling channels - where customers interact with and purchase your products. A store can be a physical retail location or an online storefront.

Physical Store

Brick-and-mortar retail locations. Customers visit, browse, and purchase in person.

  • Operating hours
  • Physical address
  • POS integration
  • In-store pickup

Online Store

E-commerce storefronts. Customers browse digitally and purchase online.

  • 24/7 availability
  • Domain/URL
  • PLP/PDP configuration
  • Delivery/shipping

How It Works: Item → Store → Customer

Items exist in Item Master. Stores configure how those items are displayed and priced. The customer sees the result.

ITEM MASTER
Product data
name, images, specs, HSN
→
STORE CONFIG
Display rules
price list, visibility, offers
→
CUSTOMER VIEW
Final display
PLP card, PDP page

Customer Visits Store (Online)

  1. Customer opens store URL
  2. System reads item's show_in_store flag
  3. System reads store's price list → calculates price
  4. System reads store's tax config → inclusive/exclusive
  5. System reads item's commodity params → specs
  6. PLP/PDP rendered with all data combined

Customer Visits Store (Physical)

  1. Customer enters physical location
  2. Staff uses POS system
  3. System reads store's price list → calculates price
  4. System reads store's tax config → receipt format
  5. System checks inventory availability
  6. Sale processed, receipt generated
Key Insight Items don't know which store they're in. Stores pull item data and apply their own configuration (pricing, display rules, offers). The same item can appear differently in different stores.
Store vs Inventory Stores are selling channels, not storage locations. Inventory management (stock levels, transfers, warehouses) is handled in a separate Inventory module. Stores define where and how you sell, Inventory defines what stock you have.
